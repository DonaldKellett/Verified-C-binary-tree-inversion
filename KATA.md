## Task

Consider the following C program `invert.c`:

```c
#include <stddef.h>

struct bt {
  unsigned value;
  struct bt *left;
  struct bt *right;
};

struct bt *invert(struct bt *p) {
  struct bt *t;
  if (!p)
    return NULL;
  t = invert(p->left);
  p->left = invert(p->right);
  p->right = t;
  return p;
}

int main(void) { return 0; }
```

We are only interested in `struct bt` and `struct bt *invert(struct bt *)` here. The former describes a binary tree structure and the latter function reverses the given binary tree in-place and returns it. In this Kata, we will use [VST](https://vst.cs.princeton.edu/) to formalize a specification for `invert()` and prove it correct.

## Our functional model

The first step in verifying a C program is to relate it to some functional (mathematical) model and prove properties of that model. Here, we first define a binary tree ADT in Coq:

```coq
Inductive bt (A : Type) : Type :=
| bt_leaf : bt A
| bt_node : A -> bt A -> bt A -> bt A.

Arguments bt_leaf {A}.
Arguments bt_node {A} _ _ _.
```

Oftentimes, the leaf node of a binary tree is defined to contain an element of type `A`, but we do not do that here since we can have an empty binary tree `NULL` in C and our model should reflect that.

Next is our inversion function:

```coq
Fixpoint bt_invert {A : Type} (tree : bt A) : bt A :=
  match tree with
  | bt_leaf => bt_leaf
  | bt_node x l r => bt_node x (bt_invert r) (bt_invert l)
  end.
```

Recall that we should establish basic properties of our mathematical model to confirm that we are modelling the right thing. What properties should we expect `bt_invert` to have? An obvious one is that `bt_invert` is an involution, i.e. applying it twice to any given tree returns the original tree:

```coq
Lemma bt_invert_involution : forall (A : Type) (tree : bt A),
  bt_invert (bt_invert tree) = tree.
```

But that is not enough as the identity function also satisfies this property. To further confirm that `bt_invert` does what we want, we note that the in-order traversal of an inverted tree is precisely the reverse of that of the original.

```coq
Fixpoint bt_inorder {A : Type} (tree : bt A) : list A :=
  match tree with
  | bt_leaf => []
  | bt_node x l r => bt_inorder l ++ [x] ++ bt_inorder r
  end.

Lemma bt_invert_inorder : forall (A : Type) (tree : bt A),
  bt_inorder (bt_invert tree) = rev (bt_inorder tree).
```

## Tying the functional model to our C program

First, we create a "typedef" `t_bt` for the VST representation of the type `struct bt` for convenience:

```coq
Definition t_bt := Tstruct _bt noattr.
```

Now, similar to the suggested reading, we define a separation logic predicate to describe the concept that the address `p` in memory is a binary tree in C that represents the binary tree in Coq `sigma`, itself a list of C "`val`ues".

```coq
Fixpoint btrep (sigma : bt val) (p : val) : mpred :=
  match sigma with
  | bt_node x l r => EX y : val, EX z : val,
    data_at Tsh t_bt (x,(y,z)) p * btrep l y * btrep r z
  | bt_leaf => !! (p = nullval) && emp
  end.
```

It is also desirable to prevent `simpl` from automatically unfolding `btrep`.

```coq
Arguments btrep sigma p : simpl never.
```

Here is the specification for `invert()`: given an input pointer `p` whose memory contents are modelled by the binary tree `sigma`, the return value is some pointer `q` whose memory contents are modelled by the inverted tree `bt_invert sigma`.

```coq
Definition invert_spec : ident * funspec :=
 DECLARE _invert
  WITH sigma : bt val, p: val
  PRE [ tptr t_bt ]
     PROP () PARAMS (p) SEP (btrep sigma p)
  POST [ (tptr t_bt) ]
    EX q:val,
     PROP () RETURN (q) SEP (btrep (bt_invert sigma) q).
```

Your task is to prove this specification. Normally, we would also define a specification for `main()` and tie all the proofs together, but we'll skip that for the purposes of this Kata.
